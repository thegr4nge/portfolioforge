**engines/contribution.py:** Create with a single function `build_contribution_array`:

```python
def build_contribution_array(
    years: int,
    regular_amount: float = 0.0,
    frequency: ContributionFrequency = ContributionFrequency.MONTHLY,
    lump_sums: list[LumpSum] | None = None,
) -> np.ndarray:
```

- Returns 1D numpy array of shape (years * 12,) with per-month contribution amounts
- Converts weekly to monthly: `amount * 52 / 12`
- Converts fortnightly to monthly: `amount * 26 / 12`
- Monthly: amount as-is
- Fills entire array with the monthly equivalent for regular contributions
- Adds lump sums at their specified month index (convert 1-based to 0-based: `month - 1`)
- Validates lump sum month is within range (1 to years*12), skips out-of-range silently

**engines/montecarlo.py:** Update `simulate_gbm` signature to accept both the old `monthly_contribution: float = 0.0` AND new `contributions: np.ndarray | None = None` parameter.

Backward compatibility logic (Pattern 4 from research):
- If `contributions` is provided and not None, use it directly
- Else if `monthly_contribution > 0`, build a uniform array from it: `np.full(years * 12, monthly_contribution)`
- Else no contributions (use the fast vectorised path)

The iterative loop for contributions should use the contributions array:
```python
paths[:, 0] = (initial_value + contributions[0]) * growth[:, 0]
for t in range(1, n_steps):
    paths[:, t] = (paths[:, t - 1] + contributions[t]) * growth[:, t]
```

This is the beginning-of-period convention already established in the existing code.
